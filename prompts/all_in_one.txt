TASK
==========

Your job is to detect tampering in the document and produce a strict forensic report.

You must do three things:
1. Inspect each DTD-flagged region and decide whether it is real tampering, what type it is, and how to ground it.
2. Search the rest of the document for tampering that DTD may have missed, especially clean semantic substitutions and logical value changes.
3. Output a final report in the exact format specified below.

DTD is a strong prior, but it does not classify tampering type. A flagged region may be:
- Visual Clumsy: visible blur, smearing, redaction, patching, obscuration, compression artefacts.
- Logical Fraud: a number, date, statistic, ID, or value has been altered but renders cleanly.
- Semantic Subtle: a clean word or short phrase has been substituted in a way that contradicts surrounding text.

DTD may also false-alarm on math-heavy regions, dense numeric tables, logos, seals, and similar graphic content. Do not treat every DTD region as tampering without inspection.

==========
GLOBAL HEURISTIC — IS DTD MISFIRING ON THIS DOCUMENT?
==========

Before walking through the regions, look at the document AS A WHOLE:

* **DTD trustworthy** (default): plain prose / form / memo / receipt / business document with a small or moderate number of DTD regions located on or near text content. Trust DTD strongly. Bias toward keeping every region; spend your effort deciding the TYPE (Visual / Logical / Semantic), not whether to keep it.

* **DTD partially misfiring** (math-heavy or scanned-table-heavy documents): the document contains substantial mathematical content (equations, formulas, summation signs, fractions, integrals, scientific notation, dense numeric tables) AND DTD has produced an unusually large number of regions concentrated on those areas. In this mode, do NOT trust DTD blindly on the math/table regions; verify by inspection and keep only those that show genuine tampering artefacts.

State your global judgment ("DTD trustworthy here" / "DTD misfiring on math content" / "mixed") at the start of your reasoning so it is auditable.

==========
WHEN TO KEEP A DTD REGION (default = keep)
==========

Given DTD's 0.66 IoU baseline, the default is KEEP. Drop a region only when one of the explicit drop rules below applies. The cases below describe what the region is likely to be — they help you classify the TYPE (Visual / Logical / Semantic), not whether to keep it.

CASES STRONGLY FAVOURING KEEP:

A. **Region inside a paragraph or between words.** A DTD region sitting inside running prose or between words, with surrounding clean text, is very likely tampered text. → most often **Visual Clumsy** if the artefact is visible, **Semantic Subtle** if the substituted word is visually clean but contextually wrong, **Logical Fraud** if a numeric value was changed.

B. **Blurred / JPEG-distorted text.** Blur, smearing, or heavy JPEG-compression artefacts inside a DTD region. → **Visual Clumsy**.

C. **Solid rectangle / white-out / black-out.** Solid coloured rectangles, white empty patches, black/grey bars over content. → **Visual Clumsy**.

D. **Empty area in the middle of running text.** A clear empty gap that breaks the natural flow of words. → **Visual Clumsy** or **Logical Fraud** (removed value).

E. **Blurred logo / stamp / seal** while surrounding document is sharp. → **Visual Clumsy**.

F. **Numeric value where the digits look clean.** Tightly-bounded number, date, percentage, total, page number, or statistic, no visible artefact. → **Logical Fraud**.

G. **Single word or short phrase, visually clean, in running prose** that contradicts surrounding context. → **Semantic Subtle**.

==========
WHEN TO DROP A DTD REGION (rare; only the cases below)
==========

DROP RULES (the only valid reasons to remove a DTD region):

H. **Tiny, weak, irregular shape**, mostly blue-green on the heatmap rather than red/yellow, AND not approximately rectangular. → DTD noise, drop.

I. **In document margins / blank borders, far from any content,** AND visually empty. → low-quality false positive, drop.

J. **Math content in a math-heavy document with massive DTD activation,** AND the formula renders cleanly. → DTD misfiring on math glyph statistics, drop.

If a DTD region does not match any of H/I/J — KEEP IT. Do not invent additional reasons to drop. The penalty for dropping a true positive is severe.


==========
TYPE CLASSIFICATION (this is YOUR primary cognitive work)
==========

Every kept region (whether from DTD or found independently) must be assigned EXACTLY ONE of three short types:

* **Visual Clumsy** — visually obvious manipulation: blur, pixelation, smearing, hard-edge redaction blocks, visible overpainting, copied blocks, clear obscuration. The evidence is the visual artefact itself.

* **Logical Fraud** — a specific value (number, date, statistic, page number, year, count, name, ID) has been altered, creating factual inconsistency or internal contradiction. The digits or letters may render fine; the evidence is the value being wrong relative to the document's own logic.

* **Semantic Subtle** — a word or short phrase was replaced with a visually clean substitution that creates a semantic contradiction with surrounding text. The evidence is the contradiction between the substituted word and the body text that references it.

Use Visual Clumsy only when the image itself shows the manipulation.
Use Logical Fraud when a value is wrong even if it looks clean.
Use Semantic Subtle when the text is visually clean but semantically inconsistent.

CRITICAL CLASSIFICATION RULES:

* A DTD-flagged region is NOT automatically Visual Clumsy. Many DTD regions are Logical or Semantic. Read the content, do not classify by source.
* A rectangular shape, black bar, or white box is NOT by itself enough to call something Visual Clumsy — it must show clear visual obscuration or editing artefacts.
* Use Visual Clumsy only when the IMAGE shows obvious editing artefacts.
* Use Semantic Subtle when the text is visually clean but contextually wrong.
* Use Logical Fraud when a value is numerically or factually inconsistent with the document's own internal logic.

==========
OCR USAGE
==========

OCR is for:
- locating text in the image,
- grounding your anomaly boxes,
- confirming what words or values occupy a region.

OCR confidence is **NOT a tampering signal**. Low confidence means the OCR engine struggled with the rendering; it does NOT mean the underlying text was tampered. The only legitimate use of OCR confidence is to suspect that an OCR token might be a hallucination of the OCR engine itself; in that case, verify the actual text by reading the image yourself.

Do NOT keep a region just because OCR confidence is low. Do NOT drop a region just because OCR confidence is high.

OCR errors (e.g. "StHl" instead of "STAY") are NOT tampering. Only flag what you can verify VISUALLY in the image, regardless of what OCR says. Use your own text-recognition ability to read suspicious regions; OCR is a hint, not a verdict.

If OCR seems wrong, verify visually before deciding.

==========
GROUNDING STRATEGY
==========

Coordinates must be absolute integer pixels in [xmin, ymin, xmax, ymax] order.

Choose grounding in this priority:
1. DTD bbox: if a kept DTD region tightly covers the tampered content, use it as-is.
2. OCR bbox: if DTD is missing or poorly aligned, use the OCR box or union of OCR boxes that cover the altered content.
3. Visual estimate: only if neither DTD nor OCR gives a usable box.

Always make the box as tight as reasonably possible.
