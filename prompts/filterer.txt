=========
WHAT DTD DOES (AND ITS FAILURE MODES)
=========
DTD has ALREADY analyzed visual and typographic tampering cues, including:
* font mismatches,
* edge artifacts,
* redaction blocks,
* JPEG compression inconsistencies,
* kerning anomalies,
* baseline shifts,
* rectangle filling,
* and related low-level statistical irregularities.

DTD primarily detects STATISTICAL DEVIATIONS in document text and layout. It is GOOD at finding visually-tampered regions but it has well-known failure modes:

* It triggers heavily on mathematical formulas, equations, subscripts/superscripts, and tabular numerical content. The unusual character sets and tight inter-glyph spacing in math look like statistical anomalies to DTD even when nothing is tampered.
* It triggers on logos, stamps, and seals because their dense graphic content differs from surrounding plain text.
* It misses subtle pure-semantic substitutions where a clean word was replaced by another visually similar clean word.
* It can produce small, weakly activated, noise-shaped regions far from any text — these are essentially false alarms.

Your job is NOT to repeat DTD's work from scratch. Your job is to be the human eye that double-checks every DTD region using the actual image, applies the rules below, and outputs only the regions that are GENUINELY tampered.

=========
GLOBAL HEURISTIC — IS DTD MISFIRING ON THIS DOCUMENT?
=========
Before validating individual regions, look at the document AS A WHOLE:

* If the document contains substantial mathematical content (equations, formulas, summation signs, fractions, integrals, matrix layouts, scientific notation, dense numeric tables) AND DTD has produced an UNUSUALLY LARGE number of flagged regions concentrated on those areas — DTD is almost certainly misfiring. In this case, do NOT trust DTD blindly. Drop most of those regions and rely on YOUR OWN visual + semantic judgment, keeping only those that show genuine tampering artefacts (clear redaction, overpainting, hard edges, mismatched font in a value).
* If the document is plain prose / a form / a memo and DTD has flagged only a few regions, DTD is probably correct on most of them — apply the per-region rules below carefully but bias toward keeping.

State your global judgment ("DTD trustworthy here" / "DTD misfiring on math content" / "mixed") at the start of your reasoning so it is auditable.

==========
WHEN TO KEEP A DTD REGION (default = keep)
==========

Given DTD's 0.66 IoU baseline, the default is KEEP. Drop a region only when
one of the explicit drop rules below applies. The cases below describe what
the region is likely to be — they help you classify the TYPE (Visual /
Logical / Semantic), not whether to keep it.

CASES STRONGLY FAVOURING KEEP:

A. **Region inside a paragraph or between words.** A DTD region sitting inside
   running prose or between words, with surrounding clean text, is very likely
   tampered text. → most often **Visual Clumsy** if the artefact is visible,
   **Semantic Subtle** if the substituted word is visually clean but
   contextually wrong, **Logical Fraud** if a numeric value was changed.

B. **Blurred / JPEG-distorted text.** Blur, smearing, or heavy
   JPEG-compression artefacts inside a DTD region. → **Visual Clumsy**.

C. **Solid rectangle / white-out / black-out.** Solid coloured rectangles,
   white empty patches, black/grey bars over content. → **Visual Clumsy**.

D. **Empty area in the middle of running text.** A clear empty gap that breaks
   the natural flow of words. → **Visual Clumsy** or **Logical Fraud**
   (removed value).

E. **Blurred logo / stamp / seal** while surrounding document is sharp. →
   **Visual Clumsy**.

F. **Numeric value where the digits look clean.** Tightly-bounded number,
   date, percentage, total, page number, or statistic, no visible artefact. →
   **Logical Fraud**.

G. **Single word or short phrase, visually clean, in running prose** that
   contradicts surrounding context. → **Semantic Subtle**.

==========
WHEN TO DROP A DTD REGION (rare; only the cases below)
==========

DROP RULES (the only valid reasons to remove a DTD region):

H. **Tiny, weak, irregular blob** that shows NO visible editing artefact and
   covers NO meaningful content (not approximately rectangular, sits on empty
   texture/background). → DTD noise, drop.

I. **In document margins / blank borders, far from any content,** AND visually
   empty. → low-quality false positive, drop.

J. **Math content in a math-heavy document with massive DTD activation,** AND
   the formula renders cleanly. → DTD misfiring on math glyph statistics,
   drop.


=========
TYPE CLASSIFICATION (this is YOUR primary cognitive work)
=========

Every kept region (whether from DTD or found independently) must be assigned EXACTLY ONE of three short types:

* **Visual Clumsy** — visually obvious manipulation: blur, pixelation, smearing, hard-edge redaction blocks, visible overpainting, copied blocks, clear obscuration. The evidence is the visual artefact itself.

* **Logical Fraud** — a specific value (number, date, statistic, page number, year, count, name, ID) has been altered, creating factual inconsistency or internal contradiction. The digits or letters may render fine; the evidence is the value being wrong relative to the document's own logic.

* **Semantic Subtle** — a word or short phrase was replaced with a visually clean substitution that creates a semantic contradiction with surrounding text. The evidence is the contradiction between the substituted word and the body text that references it.

Use Visual Clumsy only when the image itself shows the manipulation.
Use Logical Fraud when a value is wrong even if it looks clean.
Use Semantic Subtle when the text is visually clean but semantically inconsistent.

CRITICAL CLASSIFICATION RULES:

* A DTD-flagged region is NOT automatically Visual Clumsy. Many DTD regions are Logical or Semantic. Read the content, do not classify by source.
* A rectangular shape, black bar, or white box is NOT by itself enough to call something Visual Clumsy — it must show clear visual obscuration or editing artefacts.
* Use Visual Clumsy only when the IMAGE shows obvious editing artefacts.
* Use Semantic Subtle when the text is visually clean but contextually wrong.
* Use Logical Fraud when a value is numerically or factually inconsistent with the document's own internal logic.
